# Мануал: как создать Skill для агента (на примере Definitions)

Этот документ описывает практический процесс создания Skill для агента Cursor в проекте DKIT.  
Фокус примера: Skill, который помогает добавлять **новый тип Definition** (C#-класс, JSON, загрузка в `DefinitionsManager` проекта/подпроекта).

## 1) Что такое Skill в Cursor

Skill — это папка с файлом `SKILL.md`, где описано:
- **что** агент должен делать;
- **когда** применять этот Skill;
- **какие шаги** выполнять и в каком порядке;
- **какие ограничения** обязательно соблюдать.

Типичная структура:

```text
<skill-name>/
├── SKILL.md          # обязательно
├── reference.md      # опционально, подробная справка
└── examples.md       # опционально, примеры
```

## 2) Где хранить Skill

- Персональный (для всех проектов): `~/.cursor/skills/<skill-name>/`
- Проектный (в репозитории): `.cursor/skills/<skill-name>/`

Для DKIT, если Skill нужен команде, обычно выбирают **проектный** вариант.

## 3) Минимальный формат `SKILL.md`

В `SKILL.md` нужен YAML frontmatter и тело инструкции:

```markdown
---
name: definitions-new-def
description: Создает новый тип Definitions данных (класс, JSON и регистрация загрузки в DefinitionsManager). Использовать, когда пользователь просит добавить новый def/definition в проекте или подпроекте.
disable-model-invocation: true
---

# Definitions New Def

## Инструкции
...
```

## 4) Правила качества Skill

- `name`: только lowercase, цифры и дефисы.
- `description`: конкретно описывает **что делает** и **когда применять**.
- Делайте `SKILL.md` коротким; детали выносите в `reference.md`.
- Используйте одну терминологию: `Definition`, `DefinitionsManager`, `LoadSingle`, `LoadCollection`.
- Явно фиксируйте ограничения проекта (например, не трогать автогенерируемые `MpGenerated` файлы).

## 5) Шаблон Skill для Definitions

Ниже логика, которую стоит заложить в Skill:

1. Найти нужный контекст в модуле:
   - где лежат существующие def-классы;
   - где расположен `DefinitionsManager` нужного проекта/подпроекта;
   - как в нем организованы `loadMethods` и поля кэша.
2. Создать C#-класс нового def:
   - унаследовать от корректной базовой модели (`AbstractDefinition` или проектное исключение);
   - добавить сериализуемые поля.
3. Добавить JSON-файлы в `Resources/Definitions/...`:
   - один файл для `LoadSingle`;
   - или набор файлов/подпапок для `LoadCollection`.
4. Обновить `DefinitionsManager`:
   - поле хранения (`MyDef` или `Dictionary<string, MyDef>`);
   - метод загрузки;
   - включить метод в `LoadAll()` в корректном порядке.
5. Выполнить валидацию:
   - JSON читается и десериализуется;
   - `Id` совпадает с именем JSON-файла;
   - ссылки на другие def валидны (`TryGetValue`/`null`-проверки).

## 6) Что обязательно предусмотреть в тексте Skill

- Четкий чек-лист действий (без пропусков).
- Отдельный блок «Не делать»:
  - не менять `MpGenerated`/автоген;
  - не править запрещенные директории генерации;
  - не использовать вымышленные пути и имена.
- Критерии завершения задачи:
  - новый def доступен через менеджер;
  - JSON загружается без ошибок;
  - потребляющий код получает данные.

## 7) Рекомендуемая проверка перед использованием

Перед тем как считать Skill готовым:

- Проверьте, что `description` содержит явные триггеры (например: "новый def", "definitions", "DefinitionsManager").
- Убедитесь, что шаги покрывают both single и collection сценарии.
- Проверьте, что в примере нет ссылок на несуществующие файлы.
- Убедитесь, что Skill не требует изменения файлов генерации.

## 8) Готовый пример

См. папку:

`/.cursor/docs/cursor/skills/example/definitions-new-def/`

В ней есть:
- `SKILL.md` — готовый пример Skill;
- `reference.md` — расширенная памятка с шаблонами и контрольным списком.
