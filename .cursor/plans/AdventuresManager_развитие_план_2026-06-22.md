# План доработки AdventuresManager

Дата: 2026-06-22
Область: `AdventuresManager` (инициализация текущего приключения/сцены, реакция на state, публичное API текущей сцены)

## 1. Цель

Реализовать в `AdventuresManager` устойчивый runtime-поток:
- восстановление текущей точки приключения из сохранения;
- fallback на стартовую точку из настроек, если сохраненная точка отсутствует/невалидна;
- отслеживание изменений текущего приключения/сцены по событиям state;
- публикация события об активной сцене для подписчиков (UI/VM);
- предоставление публичных методов получения текущего контента и доступных действий.

## 2. Текущее состояние (как есть)

- `AdventuresManager` подписан на `AdventureStateLogic.StateChanged`, но пока только логирует событие.
- В state есть `AdventuresStateData.Adventures[adventureId]` с полями `AdventureId`, `SceneId`, `Parameters`.
- Явного централизованного указателя на активную точку (`current adventure/scene`) в state сейчас нет.
- В `RuleSettingsDef` есть `StartAdventure`, но нет отдельного `StartScene`.

## 3. Решения по контракту (утверждение перед кодом)

### 3.1 Где хранить активную точку

Решение зафиксировано:
- активная точка хранится в `AdventuresStateData` в полях:
  - `CurrentAdventureId`
  - `CurrentSceneId`
- `World.Parameters` не используется как указатель текущей точки; это слой глобального прогресса мира (открытые локации, события и т.п.).

### 3.2 Источник стартовой сцены

Вариант A (минимальные изменения, рекомендуется):
- стартовая сцена берется как первая валидная из `AdventureDef.StartScenes`.

Вариант B:
- расширить `RuleSettingsDef` полем `StartScene` и использовать его приоритетно.

## 4. Этапы реализации

## Этап 1. Контракт state для текущей точки

Задачи:
- добавить хранение активной пары `AdventureId/SceneId` в state;
- добавить state-action(ы) для установки текущей точки;
- расширить `StateChangeSource` значениями для навигационных изменений.

Ожидаемый результат:
- активная точка меняется только через `StateLogic.ProcessAction(...)`;
- `AdventuresManager` может однозначно читать текущую точку из state.

## Этап 2. Инициализация AdventuresManager

Задачи:
- при `Init()` читать активную точку из state;
- валидировать adventure/scene по `DefinitionsManager`;
- если точка невалидна или отсутствует:
  - брать `RuleSettings.StartAdventure`;
  - определять стартовую сцену (по п. 3.2);
  - записывать активную точку в state;
- после загрузки состояния публиковать событие об активной сцене.

Ожидаемый результат:
- приложение всегда стартует с валидной активной сцены;
- подписчики получают событие о текущей сцене сразу после инициализации.

## Этап 3. Реакция на изменения state

Задачи:
- в обработчике `StateChanged` сравнивать текущую активную пару из state с кешем менеджера;
- при фактическом изменении adventure/scene обновлять внутренний runtime-контекст и публиковать событие;
- игнорировать события, не влияющие на активную точку.

Ожидаемый результат:
- нет лишних перезагрузок сцены;
- корректная реакция на переходы, инициированные через state-actions.

## Этап 4. Публичный API AdventuresManager

Задачи:
- добавить методы получения:
  - текущего `AdventureId` и `SceneId`;
  - текущей `SceneData`;
  - текущего контента сцены с фильтрацией ограничений;
  - текущих доступных выборов/действий с фильтрацией ограничений;
- явно разделить:
  - «видимые» элементы;
  - «доступные к выполнению» элементы.

Ожидаемый результат:
- UI/VM получают единый источник данных по активной сцене;
- логика фильтрации ограничений находится в одном месте.

## Этап 5. Событийный контракт для UI

Задачи:
- ввести event в `AdventuresManager` (например, `ActiveSceneChanged`) с payload текущего состояния сцены;
- обеспечить единый формат payload для:
  - первого события после `Init()`;
  - последующих переходов.

Ожидаемый результат:
- подписчики (дальнейшая визуализация) не зависят от внутренней логики менеджера и state.

## Этап 6. Валидация и обработка ошибок

Задачи:
- обработать граничные случаи:
  - отсутствует `RuleSettings`;
  - `StartAdventure` не найден в дефах;
  - нет валидных `StartScenes`;
  - в сохранении ссылка на удаленную/переименованную сцену;
  - частично поврежден блок `Adventures` в state;
- добавить понятные логи ошибок и fallback-стратегии.
- в рамках MVP зафиксировать минимальный безопасный fallback (без детальной продуктовой политики).

Ожидаемый результат:
- менеджер не падает на невалидных данных и может восстановиться в допустимое состояние.

## Этап 7. Тесты

Минимальный набор:
- инициализация из валидного сохранения;
- fallback на стартовую точку из настроек;
- обработка невалидной точки из сохранения;
- событие после `Init()` отправляется один раз и с корректным payload;
- при `StateChanged` событие отправляется только при изменении активной пары;
- фильтрация контента и выборов по ограничениям.

Ожидаемый результат:
- покрыты основные сценарии и регрессии поведения менеджера.

## 5. Порядок внедрения (рекомендуемый)

1. Контракт state + state-actions + `StateChangeSource`.
2. `AdventuresManager.Init()` с восстановлением и fallback.
3. Событие `ActiveSceneChanged`.
4. Публичный API текущей сцены (контент/выборы).
5. Обработка граничных случаев и логирование.
6. Тесты.

## 6. Definition of Done

- `AdventuresManager` после старта всегда имеет валидную активную сцену.
- При изменении активной точки через state-actions подписчики получают событие обновления сцены.
- Публичные методы менеджера возвращают корректно отфильтрованные контент и действия.
- Поведение покрыто тестами на ключевые сценарии.
- Нет ручных правок/создания `.meta` файлов.
