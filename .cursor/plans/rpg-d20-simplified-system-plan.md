# План: упрощенная d20-система для модуля RPG

## Контекст и цель

Нужно развить `Assets/Modules/RPG` от data-contract каркаса до рабочего рантайма приключений, где:
- игрок управляет одним персонажем или группой до 4 персонажей;
- приключения описываются JSON-файлами;
- каждое сообщение/сцена задает контент для UI (текст, описание, картинки, варианты действий);
- выборы могут вызывать проверки d20, простые бои и переходы к следующему сообщению по ID.

Текущий статус модуля:
- модели `AdventureData`, `SceneData`, `ChoiceData`, `ChoiceActionData` есть;
- `ChoiceActionData` использует контракт `Params` (`Strings/Ints/Bools/Floats`);
- `ChoiceActionType` содержит базовый набор значений;
- реализованы `ChoiceActionExecutorFactory`, `IChoiceActionExecutor`, два executor-а (`GoToScene`, `ModifyVariable`);
- объявлены интерфейсы `IAdventureFlowController`, `IRpgVariablesController`;
- `AdventuresManager` и `StateManager` не реализованы;
- `PlayerData` и `AdventureStateData` пустые.

## Ограничения и допущения

- Требуется максимально простая, предсказуемая и контентно-ориентированная система.
- Контент должен собираться геймдизайнером через JSON без правок кода.
- Бои в упрощенной форме: без полноценной тактической карты, с минимальным числом правил.
- Внутри приключения переходы идут по `sceneId`.

## Зафиксированные решения по формату (после обсуждения)

1. Статы/параметры персонажей и мира хранятся в состоянии в виде словарей:
   - `Dictionary<string, int>`
   - `Dictionary<string, string>`
   - `Dictionary<string, bool>`
   - опционально `Dictionary<string, float>`
2. Доступ к статам/параметрам из adventure JSON всегда идет по строковому ключу.
3. `AdventureData` остается единицей описания приключения/события, `SceneData` — единицей показа контента и выбора.
4. `NotClearScene` используется как режим накопления нарратива в UI (append).
5. Для встраивания одного приключения в другое вводится стек приключений (push/pop контекстов).
6. Карта/локации выносятся в отдельный world-слой JSON, чтобы разделить глобальную навигацию и локальные сцены.
7. Стартовый хаб игры: Таверна (создание персонажа + выбор доступного приключения).
8. Выполнение `ChoiceActionData` — через фабрику executors (Zenject `DiContainer`), без передачи параметров в `Execute()`:
   - фабрика парсит `Params` при создании executor-а;
   - executor хранит готовые данные и ссылки на контроллеры;
   - `Execute()` вызывается без аргументов и без return.

## Варианты реализации

### Вариант A — Минимальный (быстрый запуск)

Идея:
- использовать существующие классы как основу;
- добавить минимальный набор типов контента/действий;
- хранить d20-логику прямо в действиях `ChoiceActionData`.

Плюсы:
- быстрый time-to-market;
- мало новых сущностей.

Минусы:
- высокое сцепление между контентом и логикой;
- сложнее расширять бои и проверки;
- выше риск "магии индексов" в `StringValues/IntValues`.

Когда выбирать:
- если нужно быстро выпустить вертикальный срез и проверить UX/нарратив.

### Вариант B — Рекомендуемый (баланс простоты и расширяемости)

Идея:
- ввести явный runtime-слой поверх DTO:
  - `AdventuresManager` (оркестратор шага),
  - `ChoiceActionExecutorFactory` + `IChoiceActionExecutor` (выполнение действий),
  - `RulesService` (d20 проверки),
  - `CombatService` (упрощенный бой),
  - `AdventureRepository` (загрузка JSON).
- оставить JSON декларативным, а правила и валидацию вынести в сервисы.

Плюсы:
- читаемая архитектура;
- проще писать тесты;
- удобно расширять новые типы действий и контента.

Минусы:
- чуть больше стартовой разработки, чем в варианте A.

Когда выбирать:
- как основной путь для production-решения.

### Вариант C — Расширяемый "мини-DSL"

Идея:
- сделать JSON-команды как отдельный DSL-комплект (условия, вычисления, события, scripted steps);
- интерпретатор команд выполняет граф приключения.

Плюсы:
- максимальная гибкость для контента;
- минимум кодовых изменений при добавлении новых сценариев.

Минусы:
- выше сложность движка и отладки;
- сложнее валидировать контент;
- дороже в сопровождении.

Когда выбирать:
- если заранее известно, что механики будут быстро расти и сильно варьироваться.

## Целевая модель (рекомендуется вариант B)

## 1) Структура приключения (JSON)

Предложение по формату:
- `adventureId`, `title`, `description`, `tags`;
- `startNodeId`;
- `nodes: []`, где каждый узел содержит:
  - `id`;
  - `content: []` (элементы UI);
  - `choices: []` (варианты действий);
  - optional `onEnter: []` (автодействия при входе).

Контент узла:
- `Text` — основной текст;
- `Description` — дополнительное описание;
- `Image` — ссылка на спрайт/адресуемый ресурс;
- `Separator`/`Hint` — опционально для UI.

Вариант выбора:
- `id`, `text`, `description`;
- `visibility`/`requirements` (показывать или скрывать);
- `effects: []` (действия при выборе);
- `nextNodeId` (быстрый переход для простых кейсов).

## 2) Набор упрощенных d20 механик

Минимум правил:
- Проверка: `d20 + modifier >= DC`.
- Преимущество/помеха: бросить 2d20 и взять max/min.
- Крит. успех: натуральная 20 (опционально автоуспех).
- Крит. провал: натуральная 1 (опционально автопровал).

Навыки (минимальный набор):
- `Might`, `Agility`, `Wits`, `Presence`.

Базовые статы персонажа:
- `hp`, `maxHp`, `armorClass`, `attackBonus`, `damageDie`, `skills`.

Партия:
- до 4 персонажей;
- активный персонаж для проверки задается в действии (или по умолчанию лидер).

## 3) Упрощенный бой

Цель:
- короткие текстовые столкновения в 1-3 раунда.

Логика:
- инициатива (по желанию: фиксированная или d20);
- ход игрока: атака/защита/спецдействие;
- сравнение попадания: `d20 + attackBonus >= armorClass`;
- урон по `damageDie` (+ модификатор опционально);
- бой завершен, когда одна сторона побеждена;
- результат боя выставляет флаги/переходы.

Контентный контроль:
- в JSON описывается encounter (`enemies`, `reward`, `onWin`, `onLose`);
- движок выполняет бой и возвращает итоговый `CombatResult`.

## 4) Слой состояния

`StateData` расширить:
- `PlayerPartyData Party` (1..4);
- `AdventureRuntimeData Adventure`:
  - `CurrentNodeId`,
  - `VisitedNodes`,
  - `Flags` (bool/int/string),
  - `Variables` (int/float),
  - `ActiveCombat` (optional snapshot).

Сохранение:
- сериализация в локальный save-slot;
- детерминированный seed для воспроизводимости бросков (опционально).

## 5) Действия (ChoiceActionType), контракт Params и executors

### Enum `ChoiceActionType` (объявлено)

- `GoToScene`, `SetFlag`, `ModifyVariable`
- `SkillCheck`, `StartCombat`
- `ApplyDamage`, `Heal`, `GrantItem`

### Контракт `Params`

- `Params.Strings`, `Params.Ints`, `Params.Bools`, `Params.Floats`
- для каждого `ChoiceActionType` — обязательные именованные ключи (таблица контракта, TBD)
- legacy `StringValues/IntValues` не поддерживается

### Фабрика и executors (реализовано частично)

Паттерн:
1. `IChoiceActionExecutorFactory.Create(ChoiceActionData)`
2. `ChoiceActionExecutorFactory` валидирует `Params` через `GetRequiredString/Int/Bool/Float`
3. `DiContainer.Instantiate<TExecutor>(parsedParams)` — executor создается с готовыми данными
4. `executor.Execute()` — side-effect через контроллер/сервис

| Type | Executor | Required Params | Controller |
|---|---|---|---|
| `GoToScene` | `GoToSceneChoiceActionExecutor` | `Strings.sceneId` | `IAdventureFlowController` |
| `ModifyVariable` | `ModifyVariableChoiceActionExecutor` | `Strings.key`, `Ints.delta` | `IRpgVariablesController` |

Остальные типы — добавить executor + case в фабрике.

   - существование `startNodeId`;
   - уникальность `node.id`;

   - корректность action payload.
6. Зарегистрировать в Zenject installer фабрику и контроллеры.

Результат этапа:

## Этап 2 — Runtime-движок приключения

1. Реализовать `AdventuresManager` как фасад:
   - загрузка приключения;
   - переход к узлу;
   - выдача доступных выборов;
   - применение списка `ChoiceActionData` через `IChoiceActionExecutorFactory`.
2. ~~Реализовать фабрику executors~~ — базовая версия готова; расширить на все `ChoiceActionType`.
3. Реализовать `IAdventureFlowController`, `IRpgVariablesController` и остальные контроллеры.
4. Реализовать `RulesService`:
   - d20 броски;
   - skill checks;
   - advantage/disadvantage.
5. Реализовать `StateManager`:
   - инициализация/чтение/сохранение `StateData`;
   - работа с флагами и переменными.
6. Добавить журнал событий шага (для UI и дебага).

Результат этапа:
- рабочий цикл "показ сцены -> выбор -> действия -> переход".

## Этап 3 — Упрощенный бой

1. Ввести `CombatService` и модели encounter/бойцов.
2. Реализовать боевой цикл в текстовом режиме.
3. Связать `StartCombat` с переходами `onWin/onLose`.
4. Добавить ограничение на сложность encounter для MVP (например, до 3 врагов).

Результат этапа:
- контентный бой управляется из JSON и влияет на состояние/сюжет.

## Этап 4 — Интеграция с UI

1. Определить выходной `NodeViewModel`:
   - `content[]`, `choices[]`, `diceResult`, `combatSummary`.
2. Привязать к экрану приключения:
   - рендер текста/описания/картинки;
   - список доступных действий;
   - показ результата броска и последствий.
3. Добавить fallback-поведение для ошибок контента (без падения клиента).

Результат этапа:
- UX-цикл приключения полностью замкнут.

## Этап 5 — Тесты и инструменты для контента

1. Unit-тесты:
   - проверки d20;
   - переходы по узлам;
   - обработка действий;
   - бой (win/lose).
2. Контентные тесты (data integrity):
   - проверка JSON-папки с приключениями в CI.
3. Debug-инструменты:
   - принудительный seed;
   - принудительный результат броска (для QA).

Результат этапа:
- предсказуемое поведение и быстрая диагностика контентных ошибок.

## Риски и как снизить

- Слишком гибкий формат действий -> ввести typed payload и строгую валидацию.
- Рост сложности боя -> зафиксировать MVP-бой и не расширять до прохождения пилота.
- Контентные ошибки в ссылках -> линтер JSON в CI + локальный валидатор.
- Расхождение UI и движка -> единый `NodeViewModel` контракт.
